What is Spec-Driven Development?

Spec-Driven Development (SDD) is a methodology that treats specifications as executable blueprints that continuously guide AI-assisted implementation. Instead of feeding natural-language prompts into an AI and hoping for the best, SDD creates durable, version-controlled artifacts that serve as the single source of truth throughout the entire development lifecycle.

💡

The core insight: Prompts are ephemeral — they lose context between sessions, vary by developer, and leave no trace. Specifications are durable — they persist across sessions, enforce consistency, and compound knowledge over time.

The Problem

When teams use AI to generate code from ad-hoc prompts, several problems emerge:

  • Architectural drift — Different developers prompt differently, producing inconsistent patterns across the same codebase
  • Context loss — Each new AI session starts from zero. Decisions made yesterday are forgotten today
  • No traceability — There's no artifact connecting a requirement to its implementation. When something breaks, there's no spec to trace back to
  • Compounding rework — Without upfront thinking, teams discover gaps mid-implementation, leading to costly rewrites

The Solution

SDD inverts the workflow. Instead of prompt → code → hope, you follow spec → plan → implement → update spec. The specification becomes a living document that:

  • Guides every AI session — New conversations read the spec, not a fresh prompt
  • Captures decisions — Architecture choices, gotchas, and trade-offs are recorded permanently
  • Enables parallelism — Multiple developers (or AI agents) can work from the same spec without diverging
  • Stays current — Specs are updated during implementation, so they always reflect reality

Specs vs. Prompts

Ad-hoc PromptsSpec-Driven Development
LifespanDies when the session endsLives in version control forever
ConsistencyVaries by developer and moodSingle source of truth for the team
ContextResets every conversationAI reads the spec — full context in seconds
KnowledgeLost after each sessionGotchas, decisions, and patterns compound
TraceabilityNo link between requirement and codeEvery implementation traces to a spec
CollaborationHard to review what was askedSpecs are reviewable, diffable, shareable

When to Use SDD

SDD delivers the most value for complex, multi-faceted features. It's not needed for every change.

Use SDDSkip SDD
Feature touches 3+ screens or flowsIsolated bug fix
Multiple user types with different permissionsCopy change or styling tweak
Cross-team handoff needed (QA, PM, design)Single-file refactor
Feature migrated from another platformAdding a field to an existing form
Architecture decisions that affect future workDependency update
⚡

Rule of thumb: If you'd spend more than a day implementing it, spend a few hours speccing it first. The documentation + planning phases pay for themselves by eliminating mid-implementation surprises. Jira-ready documentation is synthesized automatically from the final specs after implementation.

Our Workflow

We are consolidating two apps (React web + native iOS) into a single Expo codebase sharing the same backend (squire-api). Every feature goes through three phases: Documentation, Planning, and Implementation.

⏰

Time investment: Expect the documentation + planning phases to take a full day or more for medium-to-large features. This is intentional — spending time thinking early prevents costly rework later.

Quick Reference: Complete Workflow

DOCUMENTATION · Analyze feature → identify sub-flows → list tickets · Create empty Jira tickets in Epic · Validate ticket list against Figma designs · /feature-spec <feature> <sub-flows> → generates one _spec.md per flow · Review technical specs, request changes · Design coverage matrix → map every screen/state to Figma frames · Write final specs using interview prompt → refine _spec.md files · /validate-spec <feature> → Figma, API, permissions, upload-readiness PLANNING · /implementation-plan <feature> → phased plan with Phase 0 + shared code reuse · Review & adjust plan IMPLEMENTATION · For each phase: · New conversation → paste Implementation Kickoff prompt · Claude implements → use sub-agents for parallel work · Iterate (play, find bugs, fix) → paste Iteration prompt · Update plan + specs → paste Phase Completion prompt · /review-pr → /create-pr · /upload-jira <feature> <ticket mappings> → synthesize Jira content from specs & upload

Phase 1: Documentation Documentation

1
Feature Discovery & Ticket Decomposition

Before writing any specs, analyze the entire feature to identify sub-flows that deserve their own Jira ticket.

How to decompose:

  • Look at the feature in Web Commander and the new Figma designs
  • Identify distinct screens, flows, and user interactions
  • Ticket sizing rule: 1 ticket = 1 independently demo-able flow. Each ticket gets its own spec file and its own PR. Even if two flows share UI (e.g., Create Service and Edit Service share the same form), they should be separate tickets if the logic differs.
  • Consider what a reviewer can reasonably review in one PR
⚠️

One ticket = one spec file. If a sub-feature within a flow is complex enough to need its own acceptance criteria and testing (e.g., a duration picker with custom wheel behavior, a multi-select tax bottom sheet), it should be its own Jira ticket and its own spec file. When in doubt, split — it's easier to merge two small PRs than to review one massive one.

Example (Service Management):

  • Create Service
  • Edit Service
  • Delete Service
  • Manage Categories
  • Optional Staff Assignments
  • Service List
  • Toggle Service Enabled/Disabled
  • Manage Service Order (drag & drop)
  • Reset Service Order
  • Localized Service Names
  • Export Services CSV (web only)
2
Create Empty Jira Tickets
  1. Create empty tickets inside the corresponding Epic
  2. Validate against the Figma designs that these are the only tickets needed
  3. Important: If a flow deserves more detail (e.g., "filters" within "service list"), that's a separate ticket. Keep tickets focused.
3
Document Existing Apps

Use /feature-spec to reverse-engineer each sub-flow from the existing apps:

/feature-spec Services create_service, edit_service, delete_service, manage_categories, service_list, toggle_service, manage_order, reset_order, localized_names, export_csv

This produces one _spec.md file per flow — a technical spec for engineers that serves as the single source of truth:

docs/services/
├── create_service_spec.md
├── edit_service_spec.md
├── delete_service_spec.md
├── manage_categories_spec.md
├── service_list_spec.md
├── toggle_service_spec.md
├── manage_service_order_spec.md
├── reset_service_order_spec.md
├── localized_service_names_spec.md
└── export_services_csv_spec.md
💡

Tip: Ask Claude to launch sub-agents for parallel documentation when documenting multiple sub-flows. This significantly speeds up the process.

After all specs are created, review them and request changes if needed.

4
Write Final Specs (Interview-Driven)

For each ticket, send Claude a single message containing these three things:

1. Figma links for this flow

Walk through the flow as if you were a user and link the corresponding Figma frame for every screen and interaction state. Be specific — link each platform variant (web vs mobile) and each state (default, hover, error, empty).

If a flow has no Figma designs, flag it and use /generate-ui instead.

Example: Edit Service ticket

Edit Service form:

  • Mobile form layout: Figma — Edit Service Mobile
  • Web form layout: Figma — Edit Service Web
  • Category dropdown opens a bottom sheet on mobile: Figma — Category Bottom Sheet
  • Duration picker uses a scroll wheel: Figma — Duration Picker
  • Tax selection is a multi-select bottom sheet: Figma — Tax Selection
  • Save button disabled state: Figma — Save Disabled

2. Flow description & functional requirements

Merge the existing app documentation (from Step 3) with the Figma designs. Describe what the user sees, what happens on each action, validations, permissions, and platform differences for this specific ticket.

3. The Interview Prompt

Paste the Interview Prompt at the end. Claude will read everything you wrote, then interview you in depth about technical implementation, UI/UX tradeoffs, roles & permissions, edge cases, and more — asking only non-obvious questions. It keeps going until every ambiguity is resolved.

💡
Many answers will be "Check squire-app and match its behavior" — that's fine, it ensures alignment. Once the interview is complete, Claude writes the final spec to its file.

After the interview: Read the final spec and verify it makes sense. Repeat this step for each remaining ticket.

5
Validate Specs

Before moving to planning, run /validate-spec on all specs:

/validate-spec services

This cross-checks:

  • All Figma links are valid and accessible
  • All API endpoints exist in @squire/api or src/api/
  • All three user types (Brand, Shop, Barber) are covered in every spec
  • Permissions match squire-app exactly (prompts you for any differences)
  • Implementation plan consistency (if it exists)
  • Upload-readiness — checks that each spec has enough substantive content for /upload-jira to synthesize quality Jira fields (acceptance criteria, permissions, edge cases, etc.)
⚠️

Fix everything flagged before proceeding. Specs with 2+ missing sections are flagged as blockers — /upload-jira will produce incomplete Jira content without them.

Phase 2: Planning Planning

6
Generate Implementation Plan

Use /implementation-plan to analyze all specs and create a phased roadmap:

/implementation-plan services

This produces an implementation_plan.md with:

  • Phase 0: Shared Infrastructure (mandatory) — shared components, hooks, utilities, API methods, routes
  • Subsequent phases ordered by dependencies (list screens before forms, read before write)
  • Key architecture decisions pre-documented
  • Empty sections for gotchas and bug fixes (populated during implementation)
  • Cleanup phase at the end (mandatory)
🔍

Reuse first, create second. The plan searches these codebase locations for existing reusable code before planning new components:

  • @squire/ui — base UI components (Box, Text, Input, etc.)
  • src/components/_shared/ — shared project components
  • src/hooks/permissions/ — permission hooks
  • src/constants/ and src/utils/ — existing constants and utilities

Also identify components across phases that can be abstracted into shared pieces — if two screens need a similar card or list, plan a shared component in Phase 0 instead of building it twice.

7
Validate & Adjust the Plan

Review the generated plan:

  • Does the phase ordering make sense?
  • Are the PR sizes reasonable for review?
  • Are there any cross-feature dependencies the plan missed?
  • Do the architecture decisions align with how you want to build it?
  • Did it find and reuse existing codebase components instead of planning new ones?
  • Are there similar components across phases that should be abstracted into shared pieces in Phase 0?
⚡

This is the last chance to change course cheaply. Make adjustments now.

Phase 3: Implementation Implementation

🔄
Context Window Management

Claude's context window (1M tokens with Opus, 200k with Sonnet) is large but still fills when reading specs + codebase. Use these rules:

TaskConversation Strategy
All documentation (Steps 1–5)1 conversation
Planning (Steps 6–7)Same or new conversation
Implementation (per phase)New conversation per 1–3 phases
Bug fixing / iterationSame conversation as the phase

After each conversation: Update the implementation plan with decisions and status before closing.

8
Implement Phase by Phase

For each phase:

  1. Read & ask — Start a new conversation and paste the Implementation Kickoff prompt. Claude reads the plan + relevant specs. If it has questions, answer them. If not, it starts implementing.
  2. First pass — Claude implements the phase. Use sub-agents for independent work (e.g., multiple components in Phase 0).
  3. Iterate — Play with the feature. Make a list of bugs, design mistakes, and issues. Share them with Claude using the Iteration Prompt.
  4. Document & PR — Once the phase is stable, paste the Phase Completion prompt.
🎨

Respect the Figma designs. Do not underestimate design fidelity — exact texts, spacing, colors, icons, and interactions from the designs must be matched precisely. Designs are not suggestions; they are the spec for the UI. When in doubt, ask.

💡

Learn from previous phases. Before implementing, always review gotchas and decisions documented in earlier phases of the implementation plan. Similar issues often recur — avoid re-discovering the same problems.

⚡
Gotchas & Lessons Learned

After every implementation phase, document gotchas in the implementation plan. These are institutional knowledge for the team.

Examples from Service Management:

  • Unistyles StyleSheet.create() styles don't work when passed as ReactNode props across component boundaries → use inline styles with theme tokens
  • SectionList doesn't reliably re-render items when extraData changes → use ref-based imperative approach for swipe dismiss
  • Web dropdown menus must use createPortal to escape parent stacking contexts
  • Platform.OS checks don't work for responsive layouts → use useResponsiveDevice().isDesktop
📚

These gotchas save future developers hours of debugging the same issues.

📝
Updating Documents
🚨

Every implementation PR must include updates to:

1. Implementation plan — phase status, new decisions, gotchas, bug fixes
2. Spec documents — if any behavior changed from the original spec

This is not optional. Stale docs are worse than no docs. Specs are the single source of truth — /upload-jira will synthesize Jira content from them after implementation.

9
Upload to Jira

After all implementation phases are complete and specs are finalized, synthesize and upload Jira content:

/upload-jira services SQR-3188 service list, SQR-3189 delete service, SQR-3190 manage categories

The skill follows this process:

  1. Read sources — reads the final _spec.md files + implementation_plan.md (gotchas become testing notes, bug fixes become regression test cases)
  2. Match specs to tickets — matches each spec file to a Jira ticket by keyword overlap between filenames and ticket IDs/descriptions
  3. Synthesize — transforms technical spec content into non-technical Jira fields. API endpoints become functional descriptions, hook names become user-visible concepts, GIVEN/WHEN/THEN become testable checkboxes
  4. Check existing content — fetches current Jira ticket content and warns if fields will be overwritten, with a single confirmation for all tickets
  5. Upload — pushes synthesized content to each Jira ticket (description, acceptance criteria, testing notes, integration testing, release notes, and all custom fields)
💡

Always run after implementation. The Jira tickets should reflect the final, tested behavior — not pre-implementation guesses. Gotchas discovered during development and bug fixes become part of the testing notes and regression test cases.
